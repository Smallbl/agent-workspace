# OpenClaw 多Agent开发团队架构重构方案

> 基于 Claude Cloud Code 源码架构思想，针对现有三大痛点（超时抢活、测试假通过、Token浪费）的完整解决方案

---

## 一、Cloud Code 源码核心架构借鉴

### 1.1 Orchestrator-Worker 模式

**源码借鉴点：** Anthropic 官方多Agent Research 系统核心架构

```
用户需求 → LeadResearcher（主调度）
         ├── Subagent-1（并行搜索）
         ├── Subagent-2（并行搜索）
         └── Subagent-N（并行搜索）
         ↓ 汇总结果
    CitationAgent（校验引用）
         ↓
      最终结果
```

**核心原则：**
- Lead Agent 只做：规划、分解任务、派发、校验、汇总
- Subagent 只做：执行、返回结构化结果
- **绝对禁止：Lead 自行执行具体工作**

### 1.2 任务树逻辑与依赖管理

**源码借鉴点：** Claude Code 任务拆解的层次化结构

```
User Request
    ↓
Task Tree（任务树）
    ├── Task-1（无依赖，并行）→ Subagent-A
    ├── Task-2（依赖 Task-1）→ Subagent-B
    │       ↓
    │   Task-2.1（细分）→ Subagent-B1
    └── Task-3（无依赖，并行）→ Subagent-C
```

**关键约束：**
- 无依赖模块 → 并行派发
- 有依赖模块 → 严格按序，禁止跳步
- 任意环节失败 → 打回重做，主Agent不干预执行

### 1.3 Token 优化策略

**源码借鉴点：** Anthropic 论文 "Token usage explains 80% of performance variance"

| 策略 | 说明 |
|------|------|
| 独立上下文窗口 | 每个 Subagent 独立上下文，避免全局膨胀 |
| 结构化任务传递 | 固定格式，消除歧义，减少重复分析 |
| 必要信息过滤 | 只传递必要代码片段，拒绝传递冗余全文 |
| 失败后优先补充 | 异常时先补充信息，不盲目重试 |

### 1.4 测试验证机制

**源码借鉴点：** Claude Code 内置验证逻辑 + 多Agent Research 的 CitationAgent 校验机制

```
测试结论 = 必须附带以下证据之一：
  ✅ "通过" + 真实运行日志 + 截图/功能演示
  ❌ "不通过" + Bug详情 + 根因分析 + 复现步骤
```

**禁止出现的结论：**
- "看起来没问题"
- "应该可行"
- "测试通过"

## 二、团队架构设计

### 2.1 角色定义与禁止行为

#### 🧠 MainAgent（主调度总控）

| 属性 | 说明 |
|------|------|
| **定位** | Orchestrator / 项目调度总监 |
| **核心职责** | 需求理解、任务拆解、接口定义、任务派发、进度管控、结果校验、质量闸门 |
| **禁止行为** | ❌ 编写前端/后端/测试代码 ❌ 自行实现功能 ❌ 替代子Agent干活 |

**异常处理边界（仅限以下操作）：**
1. 重新精准描述任务
2. 补充关键定位信息
3. 细化任务粒度
4. 重试派发或更换子Agent

#### 👨‍💻 BackendAgent（后端开发）

| 属性 | 说明 |
|------|------|
| **定位** | 专业后端开发工程师 |
| **核心职责** | 后端接口开发、数据库设计、业务逻辑实现、API文档编写 |
| **禁止行为** | ❌ 修改前端代码 ❌ 跳过接口定义直接开发 ❌ 自行决定前后端联调方式 |

#### 👨‍💻 FrontendAgent（前端开发）

| 属性 | 说明 |
|------|------|
| **定位** | 专业前端开发工程师 |
| **核心职责** | 前端界面开发、组件实现、样式适配、调用后端API |
| **禁止行为** | ❌ 修改后端代码 ❌ 自行mock数据绕过联调 ❌ 跳过接口定义直接开发 |

#### 🧪 TestAgent（测试与实体验证）

| 属性 | 说明 |
|------|------|
| **定位** | 专业测试工程师 + 验收方 |
| **核心职责** | 全流程真实验证、测试用例编写、环境校验、结果复现、证据留存 |
| **禁止行为** | ❌ 结论模糊化 ❌ 无证据通过 ❌ 仅做代码审查替代真实验证 |

### 2.2 通信协议（唯一合法通信方式）

**所有角色仅通过标准化结构化任务通信，禁止自由对话式沟通。**

#### 任务传递模板（MainAgent → SubAgent）

```markdown
【任务编号】TASK-{YYYYMMDD}-{序号}
【任务标题】简洁明确的核心主题
【问题/需求描述】完整的需求背景或问题场景
【已定位信息】必填项，详细的关键信息（禁止为空）
【已知根因】精准问题原因 / 待进一步定位方向
【已尝试方案】已执行操作、结果、无效原因
【需要执行内容】明确具体工作，指令清晰无歧义
【输入输出约束】参数、格式、性能、兼容性要求
【依赖文件/代码片段】仅必要的相关内容
【预期结果】可量化、可验证的完成标准
【超时设置】{N}分钟
【重试策略】失败后自动重试 / 失败后上报主Agent
```

#### 结果返回模板（SubAgent → MainAgent）

```markdown
【任务编号】TASK-{原编号}
【执行状态】✅完成 / ❌失败 / ⚠️超时
【实际结果】与预期结果的对比
【验证证据】运行日志 / 截图 / 接口返回 / 测试报告
【遗留问题】未解决问题及原因
【建议】后续处理方向
```

## 三、任务流转流程

### 3.1 标准流水线（Cloud Code Pipeline）

```
用户需求输入
    ↓
┌─────────────────────────────────────────────────────────────┐
│ MainAgent：需求理解 + 任务拆解（任务树）                        │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 1：架构设计（仅 MainAgent 执行）                          │
│   - 输出：接口定义文档（API Contract）                          │
│   - 依赖：用户需求                                             │
└─────────────────────────────────────────────────────────────┘
    ↓
┌──────────────────┬──────────────────┐
│ Phase 2A         │ Phase 2B         │
│ 后端接口开发     │ 前端架构设计     │
│ (BackendAgent)   │ (FrontendAgent)  │
│ 并行执行         │ 依赖接口定义      │
└──────────────────┴──────────────────┘
    ↓（两者都完成后）
┌─────────────────────────────────────────────────────────────┐
│ Phase 3：前后端联调                                           │
│   - BackendAgent + FrontendAgent 协同                         │
│   - MainAgent 仅监督进度，不干预执行                          │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 4：测试验证（TestAgent）                                 │
│   - 全流程真实验证                                            │
│   - 证据留存                                                  │
│   - 通过/不通过 + 详细报告                                    │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 5：最终验收（MainAgent）                                 │
│   - 校验测试报告完整性                                         │
│   - 质量闸门审查                                              │
│   - 决定是否上线                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 异常处理流程

#### 子Agent超时/失败
```
SubAgent 超时/失败
    ↓
MainAgent 判断：
    ├─ 任务描述模糊？→ 补充信息，重新派发
    ├─ 任务粒度太粗？→ 细化任务，重新派发
    ├─ SubAgent 能力不足？→ 更换 SubAgent
    └─ 连续失败(≥2次)？→ 优化提示词，重新尝试
    ↓
绝对禁止：MainAgent 自己上手实现
```

#### 测试假通过防护
```
TestAgent 判定"通过"
    ↓
MainAgent 验证：
    ├─ 是否附带真实运行日志？❌→ 打回重测
    ├─ 是否覆盖常规/边界/异常场景？❌→ 打回补测
    ├─ 结果是否可稳定复现？❌→ 打回排查
    └─ 全部验证通过 → 接受结果
```

## 四、三大问题专项解决方案

### 4.1 子Agent超时抢活问题

**根因：** 主Agent缺乏约束机制，遇到超时直接替代执行

**解决方案：**

| 机制 | 实现方式 |
|------|----------|
| **硬性禁止规则** | MainAgent 系统提示词中明确写入：绝不自写代码，违规则自动告警 |
| **任务打回协议** | 超时 → 自动触发任务重新派发流程，MainAgent 仅修改任务描述 |
| **超时监控** | 每个任务设置独立超时时间，超时后自动暂停并通知 MainAgent |
| **替代执行拦截** | 检测到 MainAgent 执行开发动作时，立即阻断并提示 |

**示例：超时时的正确处理**

```
❌ 错误示范（当前问题）：
SubAgent 超时 → MainAgent 直接上手写代码

✅ 正确示范（重构后）：
SubAgent 超时
  → MainAgent 分析超时原因（任务描述/粒度/复杂度）
  → 补充信息，细化任务粒度
  → 重新派发（可更换SubAgent）
  → 等待结果
```

### 4.2 测试假通过问题

**根因：** 测试缺乏全流程验证机制，仅做代码审查

**解决方案：**

| 验证维度 | 验证内容 | 证据要求 |
|----------|----------|----------|
| 编译/运行 | 代码可正常执行 | 真实运行日志，无ERROR/异常 |
| 功能匹配 | 逻辑完全符合需求 | 接口调用日志 + 返回值 |
| 交互验证 | 页面可访问，流程顺畅 | 截图/功能演示 |
| 场景覆盖 | 常规+边界+异常用例 | 测试用例清单 + 执行结果 |
| 稳定性 | 结果可复现，无偶发 | 多次执行结果对比 |

**TestAgent 输出规范：**

```
【测试结论】✅通过 / ❌不通过
【验证证据清单】
  1. 运行日志：logs/xxx.log（附节选）
  2. 接口测试：POST /api/xxx → 200 OK，响应时间{XXX}ms
  3. 界面截图：screenshots/xxx.png
  4. 边界测试：已覆盖（具体列出）
【Bug详情】（如有）
  - 位置：xxx.ts:123
  - 现象：xxx
  - 根因：xxx
  - 复现步骤：xxx
```

### 4.3 Token浪费问题

**根因：** 任务描述模糊、上下文冗余、重复排查

**解决方案：**

| 策略 | 具体措施 |
|------|----------|
| **结构化任务传递** | 强制使用标准模板，必填字段缺失则禁止派发 |
| **上下文精简** | 仅传递必要代码片段，剔除冗余全文 |
| **禁止重复分析** | 已定位信息必须完整传递，禁止重复排查同一问题 |
| **SubAgent 审查权** | SubAgent 可拒绝模糊指令，要求 MainAgent 补充信息 |
| **错误优化机制** | 同一错误连续出现≥2次 → 自动优化对应提示词 |

## 五、配置与运行说明

### 5.1 Agent 注册配置

每个 Agent 需要独立配置文件：

**MainAgent 配置：**
```yaml
agent_id: main
role: orchestrator
model: MiniMax-M2.7
workspace: ~/.openclaw/workspace
forbidden_actions:
  - write_code
  - run_tests
  - modify_frontend
  - modify_backend
task_template: standardized
timeout_default: 180
```

**BackendAgent 配置：**
```yaml
agent_id: backend
role: backend_developer
model: MiniMax-M2.7
workspace: ~/.openclaw/workspace/backend
allowed_tools:
  - read
  - write
  - exec
forbidden_actions:
  - modify_frontend
  - bypass_api_contract
```

**FrontendAgent 配置：**
```yaml
agent_id: frontend
role: frontend_developer
model: MiniMax-M2.7
workspace: ~/.openclaw/workspace/frontend
allowed_tools:
  - read
  - write
  - exec
forbidden_actions:
  - modify_backend
  - mock_api_without_contract
```

**TestAgent 配置：**
```yaml
agent_id: test
role: tester_validator
model: MiniMax-M2.7
workspace: ~/.openclaw/workspace/test
verification_required:
  - compile_check
  - functional_test
  - integration_test
  - boundary_test
  - stability_test
evidence_required:
  - execution_log
  - screenshot
  - api_response
output_format: structured_report
```

## 六、架构运行异常处理预案

### 6.1 常见异常场景

| 场景 | 检测方式 | 处理策略 |
|------|----------|----------|
| SubAgent 超时 | 任务超时监控 | 任务打回，重新派发 |
| SubAgent 拒绝执行 | 收到拒绝响应 | 分析原因，补充信息 |
| 循环失败（≥2次） | 失败计数 | 优化提示词，更换策略 |
| 测试假通过 | 证据审查 | 打回补测 |
| Token 超出预算 | 预算监控 | 任务暂停，评估是否继续 |
| 依赖死锁 | 任务依赖图检测 | 强制解除依赖或并行化 |

### 6.2 熔断机制

当以下情况出现时，触发熔断：
- 同一任务失败 ≥3 次
- Token 消耗超过预算的 80%
- 检测到 MainAgent 执行开发动作

熔断后的处理：
1. 暂停当前任务派发
2. 记录异常日志
3. 通知王（人工介入）

## 七、方案总结

### 7.1 核心改进

| 问题 | 原方案 | 新方案 |
|------|--------|--------|
| 超时抢活 | MainAgent 直接替代 | 任务打回 + 禁止自执行 |
| 测试假通过 | 代码审查替代验证 | 全流程真实验证 + 证据留存 |
| Token 浪费 | 无结构化传递 | 标准化任务模板 + 上下文精简 |

### 7.2 架构优势

- ✅ **严格角色隔离**：MainAgent 永不越界
- ✅ **结构化通信**：消除歧义，减少重复
- ✅ **真实验证**：测试结果 100% 可信
- ✅ **可规模扩展**：流水线 + 并行执行
- ✅ **Token 优化**：独立上下文 + 必要信息传递

### 7.3 Cloud Code 源码核心借鉴点

1. **Orchestrator-Worker 模式** → 主控专注调度，子Agent专注执行
2. **任务树逻辑** → 层次化拆解，依赖管理
3. **Token 优化** → 独立上下文窗口，结构化传递
4. **验证机制** → 证据留存，结果可信
